Modern databases: Features you might not expect

A brief overview of features I did not expect modern databases to have, but found extremely useful.

In this article we will be using PostgreSQL as an example, but most of these features are present in many different databases.

Introduction

Whenever someone explains to you what a database is they usually talk about tables, rows columns, they might show you something like this


ID First Name Last Name Role
001 Guido van Rossum Creator
002 Barry Warsaw Core Developer
003 Brett Cannon Core Developer
004 Raymond Hettinger Contributor
005 Jakub Kramp Developer

Perhaps they show you a simple join, or a simple query and for the most part it is a good introduction because at the end of the day that's what a database is. It's just a way to store data in a structured way. The thing with modern databases, especially PostgreSQL is that they're more. Much more. And in this article I will try to get you interested in all the features Postgres brings to the table that you might not expect from the introduction you have just read.

Custom types

In order to store the data efficiently database needs to know what kind of data each column contains. This way it knows exactly how much memory to reserve for each cell. Some databases support a fixed number of types but postgres allows you to create custom data types.

Enum Types

For example in our table there is nothing preventing anyone from putting anything they want in the Role column. If we want to limit their choices to a few fixed options a custom enum type is a great solution.

CREATE TYPE role AS ENUM ( 'Creator', 'Core Developer', 'Contributor', 'Developer' );

If you know a little something about databases, you might wonder why not just use a simple check constraint here. And that's a fair point, but if you would like to reuse this in another table, you would have to write the constraint over and over again. With a custom enum type it's way easier.

Domain Types

If you want to create a constraint and use it across different tables there is a solution for that in postgres aswell. It is called domains. For example if you want a created_at column that only accepts values from a certain year, you can use domain for that.
